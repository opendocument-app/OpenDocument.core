# `tools/pdf`

Developer tooling for the PDF engine (`src/odr/internal/pdf/`). Not part of the
build — these scripts generate committed C++ source from vendored data.

## `generate_encoding_data.py`

Generates `src/odr/internal/pdf/pdf_encoding_data.{hpp,cpp}` — the three
base-encoding tables and the Adobe Glyph List used to map simple-font
`/Encoding` glyph names to Unicode. The build only compiles the generated
result, so there is no build-time dependency on Python.

Regenerate (no arguments):

```bash
python3 tools/pdf/generate_encoding_data.py
```

### Source data files

All four share one on-disk convention: `#` comment lines, blank lines ignored,
and two `;`-delimited fields per record.

| File | Contents | Format | Origin |
|------|----------|--------|--------|
| `glyphlist.txt` | Adobe Glyph List | `glyphname;CCCC CCCC` (hex UTF-16 code points) | downloaded (pinned commit) |
| `standard_encoding.txt` | StandardEncoding (ISO 32000-1 Annex D) | `HH;glyphname` (sparse; missing = .notdef) | vendored |
| `win_ansi_encoding.txt` | WinAnsiEncoding | `HH;glyphname` | vendored |
| `mac_roman_encoding.txt` | MacRomanEncoding | `HH;glyphname` | vendored |

Like `generate_cid_data.py`, the AGL is **not** vendored: the script downloads
`glyphlist.txt` from a commit-pinned
[adobe-type-tools/agl-aglfn](https://github.com/adobe-type-tools/agl-aglfn)
(`_AGL_COMMIT` in the script) into `tools/pdf/glyphlist.txt`, which is
git-ignored, on first run and reuses it afterwards. To refresh the AGL, bump
`_AGL_COMMIT`, delete the cached file, and rerun. The base-encoding tables are
fixed by the PDF spec, have no canonical download, and stay vendored.

See [`THIRD_PARTY_LICENSES.md`](THIRD_PARTY_LICENSES.md) for the provenance and
licensing of the vendored data.

## `generate_cid_data.py`

Generates the predefined-CMap tables for composite (Type0/CID) fonts —
`code → CID` per CMap plus `CID → Unicode` per character collection — used by
the legacy CJK encoding path (RKSJ/EUC/Big5/GBK/KSC).

```bash
python3 tools/pdf/generate_cid_data.py
```

Unlike the encoding data above, the input is **not vendored**: Adobe's
[cmap-resources](https://github.com/adobe-type-tools/cmap-resources) archive is
large (~14 MB unpacked), so the script downloads it (pinned by commit) into
`tools/pdf/cmap-resources/`, which is git-ignored, on first run and reuses it
afterwards.

> **Status.** The predefined *Unicode* CMaps (`Uni*-UCS2/UTF16/UTF32`) are
> handled directly in `pdf_cid.cpp` with no data tables (their codes already are
> Unicode), so for now nothing generated by this script is committed or compiled
> in. The legacy-CMap tables it emits — and the choice of how to store them
> compactly (block-encoded arrays measured at ~3.3 MB; zlib+base64 at ~855 KB) —
> are the remaining, deferred legacy-CJK half of the composite-font CMap work.
> See the PDF module's `AGENTS.md`.
